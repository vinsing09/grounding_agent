## What was built

### `grounding_agent/compare.py`

Pure-function module. No I/O.

- `ConfusionCell` (frozen) — `tp / fp / tn / fn` with derived
  `total`, `agreement`, `auto_pass_rate`, `reward_pass_rate`. Vocabulary
  pinned in the module docstring: tau-bench reward is the ground truth;
  each auto-eval dimension verdict is the prediction.
- `split_of(task_index, splits)` — resolves a task index to its split.
- `confusion_matrix(variant_data, splits)` — `{category: {split:
  ConfusionCell}}` with a synthetic `'all'` bucket per category.
  Excludes errored tasks (records with `'error'` key).
- `pass_rate_by_split(variant_data, splits)` — `{category|reward:
  {split: rate}}`. Includes `'reward'` row so the writeup can see
  agent capability per split alongside auto-eval coverage.
- `clause_citation_counts(variant_data)` — orders all clause ids by
  how often they appear in FAILED verdict `clause_refs`; tracks which
  dimensions cited each clause. Surfaces dead-weight clauses (never
  cited) and over-firing clauses (judge over-strictness).
- `Disagreement` (frozen) + `disagreements(variant_data, splits,
  kind={'all','fp','fn'})` — every dimension-level disagreement with
  enough context (`reason`, `clause_refs`) to drill into.
- `variant_overview(variant_data, splits)` — one-row summary for
  side-by-side variant comparison.

### `scripts/compare_to_reward.py`

CLI wrapper. Loads `results/{v0,v2}_results.json` + `data/tasks.json`,
calls each `compare.py` computation, renders Markdown
(`results/comparison.md`). Survives missing variants — if only v0 is
present (e.g. mid-eval), it renders the v0 section and warns about
v2. This let me dogfood the script against partial results while the
eval is still running.

### `scripts/run_eval.py`

- Reads `data/tasks.json`. Iterates `--variants` × `--splits`.
- Per-task crash safety: writes `results/{variant}_results.json`
  after every task. Re-runs cache by `task_index` (use `--force` to
  redo).
- v2 prepends `data/variants/v2_preamble.md` to `tau_bench.envs.airline.wiki.WIKI`
  and passes the result through `wiki_override`, leaving the env (user
  sim + reward function) untouched. Verified by a test in
  `tests/test_runner.py` (`test_run_task_wiki_override_replaces_agent_system_prompt`).
- Errors per-task are caught, recorded, and don't abort the run.

### `data/tasks.json`

Train = tau-bench indices 0–9 (the smoke test used 0 and 1); held_out
= 10–19. This is the BRIEF's two-tier split (the held-out is the
generalization test). Hand-authored comment explains why the split
exists.

### `data/variants/v2_preamble.md`

Three explicit rules emphasising the dimensions gpt-4o-mini under-
applies in v0 (mutation confirmation, prerequisite reads,
information grounding). Prepended to (not replacing) the policy, so
v2's prompt is strictly more constrained than v0's. Standalone file
so the diff between variants is reviewable without reading source.

## Decisions taken this chunk

- **Module split.** Computation in `grounding_agent/compare.py` (pure,
  tested); rendering in `scripts/compare_to_reward.py`. Keeps the
  module under 300 lines, lets tests pass dict fixtures without
  touching the filesystem, and lets the rendering evolve without
  retesting the math.
- **Clause-citation aggregation.** Not in BRIEF; added because the
  Day-1 smoke run revealed disagreements that trace to specific
  clauses (`fb-cancel-flights-after-use` mistagged as scope_adherence,
  `obl-confirm-action` cited on tasks where confirmation was given).
  This view lets the WRITEUP attribute disagreement to a clause or
  a judge rather than just a category.
- **`'all'` bucket synthesized at compute time.** Cheap and avoids the
  caller having to merge train+held_out by hand.
- **v2 prepends, doesn't replace.** Drop-in replacement of the wiki
  would change the policy the agent operates under, conflating
  prompt-discipline emphasis with policy edits. Prepending makes v2 a
  pure emphasis treatment.
- **Per-task crash safety in `run_eval.py`.** Eval takes ~80 minutes
  for the full 40 runs at gpt-4o-mini speeds; losing a partially-done
  run would be a half-day reset.

## Early signal (partial run on 2 tasks)

Comparison rendered against the first two v0 records reveals the
exact failure mode the framework is built to surface:

- Task 0 (reward FAIL): auto-eval marks 3/5 dimensions pass, citing
  no clauses — gpt-4o-mini judge gave a "looks fine" verdict despite
  the agent calling `book_reservation` without confirmation. The
  deterministic `tool_sequence_correctness` judge correctly flagged
  the missing `get_user_details`.
- Task 1 (reward PASS): auto-eval fails 4/5 dimensions, citing
  `fb-cancel-flights-after-use` on a flight that had not been used.
  This is a textbook judge polarity error compounded by the contract
  mistagging this clause as `scope_adherence` rather than
  `policy_compliance`.

So early data already implies the dominant disagreement is judge
quality (gpt-4o-mini being either too lenient or too aggressive in
citing clauses), with a secondary contribution from contract
mistagging. The WRITEUP will report this with the full 20-task data
once the run completes.

## Risks / things to watch

- Judges may converge on a few "always cited" clauses on tasks
  where they don't actually apply. Clause-citation aggregation
  surfaces this; if a single clause dominates citations across
  unrelated tasks, that's a smoking gun for either judge over-
  strictness or contract mistagging.
- Held-out generalization claim depends on the train/held_out split
  not leaking through clause text. We did not tune the contract on
  the held-out tasks (the contract is generated from policy.md alone,
  not from training trajectories), so the split is principled. Worth
  saying explicitly in WRITEUP.
