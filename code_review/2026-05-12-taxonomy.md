# Code review — `grounding_agent/taxonomy.py`

**Date:** 2026-05-12
**Module:** `grounding_agent/taxonomy.py` (134 lines incl. docstrings)
**Tests:** `tests/test_taxonomy.py` — 12 tests, all passing.

## What was built

A frozen, module-level tuple of six `FailureCategory` dataclasses. Each
carries `id`, `name`, `description`, `judge_dimension`, `judge_kind`
(`"semantic" | "deterministic"`), and `example`. Two helpers:
`get_category(id)` and `category_ids()`. Two import-time asserts pin the
two invariants the rest of the codebase will assume: ids are unique, count
is exactly six.

## Reconciliation with plan (`BRIEF.md` + `knowledge.md`)

- BRIEF says `taxonomy.py` defines "six failure categories, each with a
  description, example trajectory, and judge dimension name." ✅
- `knowledge.md` decisions taken: trim to six categories — done; each
  category's judge_dimension is pinned equal to its id (tested) so judge
  prompts cannot drift from taxonomy.
- BRIEF says four semantic judges + one deterministic checker. Taxonomy
  has 5 semantic + 1 deterministic = 6 categories. One semantic category
  will not have a dedicated judge prompt in `judges.py` — most likely
  `task_completion` is folded into the τ-bench reward comparison (the
  ground-truth reward already measures task completion, so a separate
  semantic judge would be redundant noise). Decision deferred to
  `judges.py`; the taxonomy is agnostic.

## Why these six (and not others)

Derived directly from policy.md's load-bearing rules:

| Policy clause | Taxonomy category |
|---|---|
| "obtain explicit user confirmation (yes) to proceed" | `confirmation_discipline` |
| "should not provide any information, knowledge, or procedures not provided by the user or available tools" | `information_grounding` |
| "should deny user requests that are against this policy" + all business rules | `policy_compliance` |
| "transfer to a human agent if and only if…" | `scope_adherence` |
| "only one tool call at a time" + all prerequisite-read patterns | `tool_sequence_correctness` |
| (implicit ground truth — did the user's goal get done?) | `task_completion` |

Cut from earlier 11-category drafts: separate "hallucination", "refusal
quality", and "instruction adherence" categories — these collapse into
`information_grounding`, `scope_adherence`, and `policy_compliance`
respectively. Six well-justified dimensions, not eleven shallow ones.

## Test rigor

- Tests assert structural invariants (count, unique ids, no empty fields,
  judge_kind constrained) not happy-path identity. Each invariant is the
  one the rest of the codebase will rely on; if it breaks, judges or the
  contract validator will silently misbehave.
- `test_required_ids_present` pins the exact six ids so downstream judge
  filenames / prompt files cannot drift without this test failing.
- `test_taxonomy_is_immutable_dataclass` proves `frozen=True` is in
  effect — the in-memory taxonomy cannot be patched at runtime by a
  test fixture or buggy import.

## Risks / things to watch

- The `judge_kind` field is consumed by nobody yet. Risk: when
  `judges.py` lands, the field's semantics might want refinement (e.g.
  "semantic_yes_no" vs "semantic_scored"). Will revisit then.
- `example` field text is human-only; not yet used as a few-shot in
  judge prompts. If we move to few-shot judges later, these become
  load-bearing strings and need a separate fixture file.
