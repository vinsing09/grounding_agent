# Code review — `grounding_agent/contract.py`

**Date:** 2026-05-12
**Module:** `grounding_agent/contract.py` (~180 lines incl. docstrings)
**Tests:** `tests/test_contract.py` — 17 tests, all passing.

## What was built

Three responsibilities in one module:

1. **Schema** — `CONTRACT_KEYS`, `CLAUSE_KEYS` declare the canonical
   shape (top-level keys + per-section required clause keys). Doc-string
   shows the JSON form a generated contract takes.
2. **Validator** — `validate_contract(contract)` raises `ContractError`
   on any structural deviation. Checks: top-level keys, agent name
   non-empty, each section is a list, every clause has its required
   keys, ids non-empty, ids unique *globally* (across sections),
   category ids are in `taxonomy.category_ids()`, prerequisite_tools is
   a list of non-empty strings.
3. **IO** — `load_contract(path)` validates on read; `save_contract(c,
   path)` validates before write (so an invalid contract never lands on
   disk; tested via `test_save_validates_before_write`).
4. **Generator** — `generate_contract(policy_text, tool_catalog, model,
   agent_name)` does the single LLM call via `litellm.completion(...,
   response_format={"type": "json_object"})`. The system prompt
   constrains the model to the exact taxonomy id list and the exact
   clause schema. Output is parsed, agent name backfilled if missing,
   then `validate_contract` gates the return.

## Reconciliation with plan (`BRIEF.md` + `knowledge.md`)

- BRIEF: "given policy.md + tools, one LLM call produces obligations +
  forbidden behaviors + tool_sequences, each tagged to a taxonomy
  category. Output committed as JSON." ✅ — `generate_contract` is that
  one call; `save_contract` writes `data/contract.json` (caller
  responsibility, via `scripts/generate_contract.py`).
- `knowledge.md` decision: "contract is generated, not hand-curated …
  removes the 'you curated favourable obligations' critique". The
  module deliberately offers no `add_clause(...)` API — generation is
  the only legal authoring path, and the validator is the only legal
  inspection path. ✅
- Taxonomy linkage: `validate_contract` calls `category_ids()` and
  rejects any clause whose category isn't in the taxonomy. If we add or
  rename a category, every existing contract file will fail loading
  until regenerated — exactly the coupling we want.

## Why these specific validator rules

Each rule corresponds to a downstream consumer's assumption:

- **Unique ids globally.** Judges may key results by clause id; a dup
  silently overwrites. Tested with both within-section and
  across-section dups.
- **Category must be known.** Judges look up the category to decide
  which dimension a finding belongs to; an unknown category produces
  uncategorized findings. Validator fails loud at the boundary.
- **`prerequisite_tools` is `list[str]`.** The deterministic checker
  iterates this list; if a generator emits a string the checker would
  iterate characters. Tested with both wrong-type and
  empty-string-entry variants.
- **`target_tool` non-empty.** Otherwise the deterministic checker
  scans for `""` in a tool list and over-matches.

## Lean-code check

- ~180 lines incl. docstrings; ≤300 target met.
- No abstractions beyond what's used. No Pydantic / dataclasses — JSON
  in, JSON out, dict validation. Adding a model layer would force every
  call site to import schemas; for a single agent-under-test that's
  weight without payoff.
- One non-obvious comment in the docstring: explains *why* the LLM
  call uses `response_format={"type": "json_object"}` — to bind the
  output to JSON rather than risk markdown-fenced text. Removing this
  hint would invite "let me parse markdown" regressions.

## Test rigor (per global rule T3)

- Fixture is realistic: clauses quote actual policy.md language and use
  the exact taxonomy ids. It is the shape the generator will emit
  against `vendor/tau_bench_airline/policy.md`.
- Each negative test mutates the fixture by exactly one field — the
  test failure tells you which schema rule fired.
- Roundtrip test (`test_save_and_load_roundtrip`) writes to a
  multi-level `tmp_path / "sub" / "contract.json"` to prove
  `parent.mkdir(parents=True)` behaves.
- `test_save_validates_before_write` proves no partial / invalid file
  ever lands on disk: after the raise, the path does not exist.

## What is NOT tested here (and why)

- **The live LLM call.** `generate_contract` hits the network and is
  non-deterministic on prompt drift; unit-testing it is brittle and
  expensive. It will be exercised once via
  `scripts/generate_contract.py`, the output committed to
  `data/contract.json`, and that committed file will be loaded by tests
  that *do* run in CI (next chunks).
- **Specific clause content.** The taxonomy already pins category ids;
  whether obligation #3 lists baggage limits before payment limits is
  not a structural concern.

## Risks / things to watch

- `litellm.completion(...)['choices'][0]['message']['content']` —
  litellm's response object also exposes `.choices[0].message.content`
  attribute access. Both work in current versions, but if litellm
  changes shape this is the fragile line. Will validate behavior
  against installed version when running `scripts/generate_contract.py`.
- The `response_format={"type": "json_object"}` flag is honored by
  OpenAI models. For Anthropic via litellm, the flag is silently
  ignored and we'd rely on prompt discipline + json.loads to fail loud.
  Acceptable; the call is run once and committed.
- We allow zero clauses in any section. If the generator returns empty
  arrays, validation passes but downstream judges have nothing to
  check. That's a generator-prompt issue, not a schema issue; will
  surface when we read the actual generated contract.
